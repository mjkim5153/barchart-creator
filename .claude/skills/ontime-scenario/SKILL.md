# ontime-scenario 스킬

사용자가 특정 편(들)을 비운항 처리하고, 나머지 스케줄만으로 정시율이 최적화된
시나리오를 만들어 저장해달라고 할 때 사용한다. 로직은 프로젝트 루트의
`build_ontime_scenario.py`(커밋된 정식 스크립트)에 구현되어 있으며, 이 스킬은 그
스크립트를 올바르게 호출하고 결과를 검증·보고하는 절차를 담당한다.

## 핵심 메커니즘 (알고리즘을 왜 신뢰할 수 있는지)

서로 다른 두 기재(Acno)가 같은 공항에 도착하는 지점을 찾아 그 뒤 체인을 통째로
맞바꾼다. `data_processor.compute_scenario_delay_adjustments()`는 Acno가 바뀐
편에 대해 새 직전편 RI+MTTT 기준으로 연결지연을 물리적으로 재계산해주므로, 이
스왑으로 (1) 비운항 때문에 끊긴 연결을 복구하고 (2) 아침에 큰 지연을 맞은 기재의
이후 체인을 깨끗한 기재 쪽으로 옮겨 지연 전파를 끊을 수 있다.

스크립트에는 실제 버그로 검증된 두 가지 안전장치가 있다 — **임의로 제거/완화하지
말 것**:
1. 같은 편명이 하루에 여러 번 재사용되는 셔틀 노선이 있으므로, 이동 대상은 항상
   Fltno + 현재 소속 Acno로 함께 매칭한다.
2. 스왑 직후 두 체인을 직접 재구성해 인접편 연결(`Arrstn==Depstn`)을 재검증하고,
   하나라도 끊기면 그 후보를 기각한다("같은 공항 도착 지점끼리는 항상 안전"이라는
   가정은 틀렸음이 실측으로 확인됨).

여기에 더해 운항 실무 관점의 4가지 제약이 후보 탐색·채택 단계의 하드 필터/리젝트로
적용된다(상세 배경·구현 위치는 `.claude/rules/ontime_optimization_rules.md` 참고,
**임의로 제거/완화하지 말 것**):
- **Curfew**: GMP/PUS는 23시 이후 이착륙 금지. 이번 조정이 새로 만든 위반은
  break와 동일하게 반드시 해결을 시도하고, 끝까지 미해결이면 저장을 중단한다(원래
  실적에 있던 위반은 로그만 남기고 저장은 진행).
- **해외공항 보존**: 연결 지점(station)이 `data_processor.KOREAN_AIRPORTS`(ICN
  포함 국내공항)가 아니면 그 지점에서의 스왑/이식 후보 자체를 만들지 않는다(국내
  공항에서는 국제선간 연결이어도 조정 가능).
- **MTTT 동적 재계산**: 원래 MTTT가 35/60이었던 국내선은 체인이 바뀌면 새 직전편의
  국내/국제 성격에 맞춰 MTTT를 60(국제선 연결)/35(국내선 연결)로 재조정한다.
- **지연 몰아주기 방지**: 어떤 편이든 실제 실적 대비 +30분을 초과하는 신규 지연을
  만드는 후보는 무조건 기각하고, 동률이면 최대 개별 지연이 더 작은 후보를 우선한다.

## 실행 절차

1. 사용자 요청에서 다음을 확인한다(이미 메시지에 있으면 재질문하지 않음):
   - **비운항 대상 편명**(콤마로 구분되는 Fltno 목록)
   - **저장할 시나리오 이름**(없으면 "`<편명목록> 비운항 (정시성 최적화)`" 형태로
     제안)
   - **기준 데이터**: 매번 최신 저장 시나리오만 쓰는 게 아니라는 점에 유의한다.
     - 사용자가 엑셀 파일을 직접 건네주거나 경로를 언급하면 그 파일 경로를
       `--source`로 사용
     - 특정 저장 시나리오를 지목하면 그 `scenario_id`를 `--source`로 사용
       (`/api/scenario-save/list` 또는 `data/scenarios/*.json`으로 이름↔id 확인)
     - 아무 언급이 없으면 `--source auto`(가장 최근 저장된 시나리오의 `original_df`,
       즉 "실적" 기준)를 사용 — **`current_df`(이전 시나리오의 조작 흔적)는 절대
       기준으로 삼지 않는다**는 점을 사용자에게 필요시 짚어줄 것(항상 순수
       비운항+재최적화만 반영된 새 시나리오를 만든다는 의미).
2. 다음 형태로 실행한다(**반드시 `run_in_background`로 실행** — 154일 전체 규모
   기준 실측 약 35~40분 소요됨(2026-07-14 검증 시 2,209초). 데이터 규모나 비운항
   대상에 따라 더 걸릴 수 있음):
   ```
   python build_ontime_scenario.py --cancel <편명1>,<편명2> --name "<이름>" [--source auto|<scenario_id>|<엑셀경로>]
   ```
3. 완료 후 로그에서 다음을 확인한다:
   - `비운항으로 인한 신규 오류(미해결)`: **0건**이어야 정상. 0이 아니면 스크립트가
     저장을 자동 중단하고 종료 코드 1을 반환한다 — 이는 실패가 아니라 정상적인
     안전장치이므로, 어떤 날짜/기재가 왜 막혔는지(출력된 오류 목록) 그대로
     사용자에게 보고한다.
   - `전체 정시율 비교`(실적/비운항 직후/최종) 3단 수치
   - `정시율 개선폭 상위 15일`, `최저 15일 개선 현황` 표
   - 저장 성공 시 마지막에 출력되는 `저장 완료: id=...` 및 메타데이터
4. 저장이 정상 완료됐다면, 임시로 서버를 띄워(`python -m uvicorn app:app --host
   127.0.0.1 --port <미사용포트>`) `/api/scenario-save/{id}/load` 로 불러온 뒤
   `/api/barchart`(연결 오류 0건 확인) + `/api/ontime`(정시율 수치가 스크립트
   로그와 일치하는지)을 대표 날짜 2~3개로 이중 검증하고, 검증 후 반드시 서버를
   종료한다(포트 점유/좀비 프로세스 방지).
5. 사용자에게 다음을 포함해 보고한다:
   - 비운항 처리된 편, 사용한 기준 데이터 출처
   - 연결 오류 해결 현황(전부 해결 / 일부 미해결 시 날짜·기재 명시)
   - 정시율 Before/After 비교(전체 + 개선폭 큰 날짜 위주)
   - **caveat 문구**: 이 결과는 MTTT 기반 이상적 재계산(그 시점에 그 기재가
     물리적으로 그 공항에 있었다면 가능한 최선)이며, 실제 크루/게이트 가용성 등은
     검증하지 않았다는 점을 항상 포함한다.
   - 저장된 시나리오 이름/ID 및 "저장/불러오기"에서 확인 가능함을 안내

## 주의사항

- `build_ontime_scenario.py` 내부의 두 안전장치와 "미해결 오류 있으면 저장 안 함"
  하드 가드는 실제 버그(편명 오매칭으로 엉뚱한 회차가 끌려감, 스왑 후 재정렬로
  1,400여 건의 연결 오류가 폭증)를 겪고 나서 추가된 것이다. 결과가 이상해 보여도
  이 로직을 우회하거나 완화하지 말고, 원인을 먼저 분석해 사용자에게 보고한다.
- curfew도 동일한 하드 가드 대상이다 — 비운항/스왑으로 새로 생긴 curfew 위반이
  끝까지 미해결이면 break와 마찬가지로 저장이 자동 중단(exit 1)된다(원래 실적에
  이미 있던 위반은 예외). 상세는
  `.claude/rules/ontime_optimization_rules.md` 참고.
- 이번 4가지 제약(curfew/해외공항 보존/MTTT 동기화/지연 캡) 추가로 후보 하나당
  체크 비용이 늘어 기존 대비 실행 시간이 더 걸릴 수 있다 — 여전히 반드시
  `run_in_background`로 실행한다.
- 실행이 오래 걸리므로 완료까지 짧은 간격으로 폴링하지 않는다(harness가 완료 시
  task-notification으로 알려준다).
- 스크립트는 항상 지정된 소스의 `original_df`(불변 실적)에서 새로 시작하고
  `current_df`는 쓰지 않는다 — 이전 시나리오의 부수적 조정을 이어받지 않는다는
  뜻이므로, 여러 비운항 요청을 "누적"하고 싶다면 이전 결과 시나리오 id를 새
  `--source`로 명시해야 한다(그래도 그 저장본의 `original_df`가 쓰이므로, 사실상
  "실적 + 이번 비운항"만 반영되고 이전 스왑 결과는 이어지지 않는다는 점을
  사용자에게 미리 안내할 것).
- `data/scenarios/`에 스모크테스트 등으로 임시 저장한 시나리오가 남지 않도록,
  검증 목적으로 직접 실행한 결과물은 작업 종료 전 삭제한다.
